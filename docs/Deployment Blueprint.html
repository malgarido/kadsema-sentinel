KADSEMA Sentinel · Engineering Handoff

Production Deployment Blueprint

How to take the SENTINEL prototype from a working interactive build to a live, multi-user emergency-operations system for the Kaduna State Emergency Management Agency.

The prototype you have is a complete, production-grade front-end with a real state model and working workflows (log → validate → escalate → respond → close; alert dispatch). It runs on browser storage. "Going live" means standing up a backend, database, authentication and real-world integrations behind that same UI. This document is the build plan.

01 Where the prototype stands

✓ Already built (reusable)

  • Full UI: command dashboard, GIS risk map, incident board, early-warning console, mobile field app
  • A clean state layer (store.jsx) with the exact actions a backend must expose
  • Working lifecycle, alert dispatch, audit log, persistence
  • Design system, component library, demo data on real LGAs

✗ Needed for production

  • Server + REST/realtime API
  • Database (PostgreSQL + PostGIS)
  • Authentication & role-based access
  • Live integrations (telecom, NiMet, NIHSA, Kobo, GIS)
  • Cloud hosting, backups, monitoring
The single most important architectural decision already made for you: every mutation in the app goes through one store with named actions (addIncident, advanceIncident, dispatchAlert). Re-point those functions at API calls and the UI becomes a real client with near-zero rework.

Start the non-code work now. Telecom short-code licensing, inter-agency MoUs (NiMet/NIHSA/security), NDPR data-protection registration, and hosting budget run in parallel with development and are usually the critical path — not the software.

07 Access control & security

Role-based access (RBAC)

  • HQ — full visibility, classification, alert issuance, inter-agency
  • Zonal — validate, assign teams, relay to HQ
  • LGA — intake, first response, escalate
  • Community — report & receive only

Security baseline

  • NDPR-compliant data handling & consent
  • Encryption in transit & at rest
  • Full audit trail (already modelled)
  • Daily encrypted cloud backups
  • Offline-capable field app for low connectivity

08 Phased rollout

Phase 0Days
Demo & buy-in. Host this prototype (Netlify/Vercel/GitHub Pages — free) as a shareable link for leadership and funders. No backend required.
Phase 10–3 months
MVP pilot, 2–3 LGAs. Stand up Supabase backend, wire incident lifecycle & auth, connect Kobo + one SMS gateway, ship the field PWA, train pilot staff. Recommended LGAs from your GIS pilot: Kaduna North, Igabi, Zaria.
Phase 23–6 months
Integrate & expand. NiMet/NIHSA live feeds, USSD short-code, GIS tile layers, siren control, statewide user onboarding.
Phase 36–24 months
Predictive & automated. AI flood/outbreak/displacement models, automated trigger→action, full 23-LGA coverage, continuous optimisation.

10 Immediate next steps

  1. Deploy this prototype to a public URL for stakeholder demos (Phase 0 — can be done today).
  2. Choose the backend approach (recommend Supabase for speed) and provision the database with the schema above.
  3. Re-point store.jsx actions to API calls — the cleanest first engineering task.
  4. Open telecom & inter-agency (NiMet/NIHSA) conversations in parallel — long lead time.
  5. Select the 2–3 pilot LGAs and recruit/ train desk officers and ward committees.
